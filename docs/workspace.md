# Workspace guide

This document explains how this repository is organized and how the build/deploy workspace is expected to behave.

## Repository layout

| Path | Purpose |
| --- | --- |
| `content/<lang>/` | Markdown page bodies for English, French, and Croatian. |
| `locales/<lang>.json` | Translated site strings, menu labels, page titles, cards, and shared copy. |
| `config/pages.json` | Page keys, slugs, enabled languages, menu order, and template selection. |
| `config/cards.json` | Card groups rendered on index/card-style pages. |
| `config/seo.json` | Canonical URL, SEO, sitemap, and related metadata. |
| `config/hero_images.json` | Hero/background image mapping by page and language. |
| `templates/base.html` | Shared HTML document shell. |
| `templates/page.html` | Generic page wrapper. |
| `templates/partials/` | Reusable HTML partials for header, footer, cards, hero blocks, and inline CSS. |
| `assets/` | Static CSS, JavaScript, image assets, and copied runtime files. |
| `tools/` | Build, validation, publishing, and deployment helper scripts. See `docs/python-tools.md`. |
| `server/` | Files that can be installed on the hosting server, such as the GitHub webhook endpoint. |
| `docs/` | Operational notes for future maintainers. |

## Generated and runtime directories

These are intentionally outside the repository or ignored by Git:

| Path | Purpose |
| --- | --- |
| `../site-dist/` | Temporary build output created by `tools/build.py`. Safe to delete. |
| `../public_html/` | Production publish target on the hosting server. Do not edit generated language roots manually. |
| `../public_html/preview/pr-<number>/` | PR preview output generated by the deploy worker. |
| `../public_html/.private/` | Server-only private deployment/contact config, logs, locks, and queues. |
| `.deploy-worktrees/` | Git worktrees used by the webhook worker for PR preview builds. |
| `tools/.venv/` | Optional Python virtual environment. |

## Build pipeline

The normal pipeline is:

```text
validate_locales.py
  -> format_hyperlinks.py
  -> build.py
  -> publish.py
```

`tools/build_and_publish.py` is the canonical wrapper for that pipeline. Production and preview behavior is controlled by destination and URL arguments.

Production-style publish:

```sh
python tools/build_and_publish.py --root . --dest ../public_html
```

Preview-style publish:

```sh
python tools/build_and_publish.py \
  --root . \
  --dest ../public_html/preview/pr-123 \
  --url-prefix /pr-123 \
  --lang-in-url \
  --write-preview-index
```

Compatibility wrappers still exist because hosting scripts may already call them:

- `tools/BUILD_AND_PUBLISH.py`
- `tools/dev_build_and_publish.py`

Prefer `tools/build_and_publish.py` for new scripts. See `docs/python-tools.md` for the full tools reference.

## Local development checklist

From the repository root:

```sh
python -m pip install -r requirements.txt
python -m compileall -q tools/
python tools/build.py --root .
```

For a full local publish test, use a disposable destination outside the repository, for example:

```sh
python tools/build_and_publish.py --root . --dest ../public_html-test
```

## Editing content safely

When editing page content:

1. Keep Markdown in `content/<lang>/`.
2. Keep translated labels and card text in `locales/<lang>.json`.
3. Keep page routing, slugs, enabled languages, and template selection in `config/pages.json`.
4. Run `validate_locales.py` or the full build before publishing.

Do not edit generated files under `../site-dist/` or `../public_html/<lang>/`; those changes will be overwritten by the next publish.

## Publishing safety

`tools/publish.py` publishes with delete semantics, so production output should be generated from a complete and valid `../site-dist/` tree.

The production publish step preserves these root-level runtime items by default:

- `preview/`
- `.private/`
- `github-webhook.php`

That means production deploys should not remove PR previews, private config, queues, logs, or the installed webhook endpoint.

## Preview safety

The webhook worker refuses to build previews from forked repositories unless `allow_preview_from_forks` is explicitly set to `true` in the private server config.

Keep this disabled unless the build environment is isolated enough to run untrusted code.
